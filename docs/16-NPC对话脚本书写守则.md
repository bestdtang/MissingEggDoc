# NPC 对话脚本书写守则

> 用途：批量生产各 NPC 的最终对话脚本。  
> 原则：角色详档 → 对话脚本（本守则：逐句）→ 树状脚本（[`18`](./18-树状对话脚本生成方法.md)：节点、分支、变量、入口 / 出口）。  
> 树状细化流程与格式见 [`18-树状对话脚本生成方法`](./18-树状对话脚本生成方法.md)；实施样章见 [`大黄-对话脚本-树状样章`](./characters/大黄-对话脚本-树状样章.md)。

---

## 16.1 脚本定位

对话脚本不是角色独白稿，也不是剧情概述，而是**可直接落进游戏对话框的逐句实施稿**。

每一行都要回答三个问题：

- 玩家此刻凭什么能说这句话？
- NPC 此刻凭什么知道 / 不知道这件事？
- 这一句是在推进信息、表现性格，还是制造轻喜剧节奏？

如果一句话不能被当前线索、当前状态或当前观察支撑，就不要让玩家说。

---

## 16.2 基础格式

每一条游戏内文本单独成行，默认对应一次点击推进。

```text
角色："台词。"
描述：动作、表情、停顿、视线、场景变化。
〔菜单〕
**A** → "玩家选项" → 进入某节点
> 〔系统注〕机制说明，不出现在游戏内。
```

格式规则：

- `角色："..."`：只写对话框里会出现的台词。
- `描述：...`：承担轻量演出、镜头、动作、表情、停顿，不另写复杂演出需求。
- `〔系统注〕`：只给策划 / 实现看，不能混成游戏内文本。
- `〔菜单〕`：用于分支入口、锁条件、节点跳转。
- 一句不要塞太多信息。长句拆成多行，让玩家点击推进。
- **说话方**：使用角色本名，或游戏内当前的伪装名（如黑猫伪装为「大树」时，揭穿前用「大树」，揭穿后改用「黑猫」）。**地点名、环境名、物件名不得作为说话方**。没有身份伪装的 NPC，无论身处何处，说话方一律用本名。

---

## 16.3 推荐文档结构

每个 NPC 对话脚本建议包含以下**四个阶段**。阶段内的节点数量和命名由该 NPC 的交互类型决定，**不要照搬其他 NPC 的子节点结构**。

### 四个阶段

**一、前置 / 尚不可交互（如有）**  
玩家触发正式对话之前，NPC 以环境台词、远距离单播、条件解锁等形式出现。若 NPC 一开始就可直接交互，省略此阶段。

**二、主交互阶段**  
核心信息推进。节点数量和命名由交互类型决定（见下方"常见类型"）。只写该 NPC 实际需要的节点，不要为填满结构而凑子节点。

**三、可选追问（如有）**  
玩家主动追问触发的推理补充。不是每个 NPC 都需要。

**四、二周目 / 回访**  
情绪收束，见 16.12。

---

### 主交互阶段：按锁点结构分类（参考，不是模板）

结构差异的本质只有一个维度：**有没有锁点，以及锁点是几个、怎么组合**。


| 类型        | 结构说明                              | 示例                         |
| --------- | --------------------------------- | -------------------------- |
| **无锁型**   | 随时可对话推进，无前置条件                     | 淑芬（直接问答，情报量少）              |
| **单锁型**   | 一个前置条件解锁主线推进，条件可以是展示证据、完成任务或状态变化  | 乌鸦（展示笔记本触发承认）、老鼠兄弟（付钱才给情报） |
| **多锁并行型** | 多个条件必须**同时**满足才触发放行；各条件之间可以任意顺序完成 | 黑猫（案情汇报线 + 薄荷鱼线同时完成才起身）    |


说明：

- 多锁并行型中每一条线本质上仍是单锁型，区别只在于"几把锁都开了才能走"。
- 同一 NPC 可以在不同节点使用不同锁类型（如先无锁闲聊，后单锁推进）。

## 16.4 玩家角色写法

玩家不是全知旁白，而是**捧哏 + 轻推理侦探**。

玩家可以说：

- 基于已获得线索的事实汇报。
- 基于眼前动作的即时观察。
- 为 NPC 搭台阶的追问。
- 轻度吐槽、确认、误会、复述。

玩家不可以说：

- 尚未被线索支撑的结论。
- 只有编剧知道的事实。
- NPC 支线未来才会揭晓的信息。
- 过度专业、过度定性的推理总结。
- **不可交互状态**（攀爬、平台跳跃、过场演出）期间的任何台词。此类阶段由 `〔系统注〕` 说明状态，NPC 可单方面播放环境台词，玩家无回应。

例：

```text
不推荐：玩家："老鼠偷了你的东西，逃跑时弄丢了。"
原因：玩家此时还不知道老鼠偷了什么，也不知道是否弄丢。

推荐：
描述：黑猫视线短暂飘向红顶屋墙缝方向，很快收回
玩家："跟老鼠有关系吗？"
```

## 16.5 NPC 信息边界

写每句 NPC 台词前，先确认这个 NPC 在此刻的认知范围。

必须区分：

- **事实真相**：策划知道的完整事件。
- **NPC 目击**：NPC 亲眼看到 / 亲耳听到的部分。
- **NPC 误读**：NPC 按自己性格理解错的部分。
- **玩家已知**：玩家此刻通过线索或对话掌握的部分。
- **待揭晓**：必须保留到后续场景或漫画收束的信息。

例：黑猫可以知道软垫被夺、乌鸦守的是石头、老鼠偷了自己的东西；但不能在进屋前知道保温箱、黄灯用途、蛋的具体位置。

一旦 NPC 提前说出待揭晓信息，就会破坏侦探节奏。

---

## 16.6 证据与台词的对应

每个关键质询都要有前置证据。

推荐写法：

```text
**★ 节点名**
*〔需持【线索A】+【线索B】〕*

玩家："基于线索能说出口的问题。"
NPC："反应。"
```

证据强度分三档：

- **硬必证据**：主线必然获得，可合并汇报，不必拆成多个节点。
- **环境依据**：证明玩家有资格问某个方向，但不能替玩家说出结论。
- **即时行为依据**：NPC 的视线、停顿、反应让玩家可以顺势追问。

避免"玩家替编剧说结论"。玩家应该先问，NPC 失态、反驳、补漏，再由对话自然推进。

## 16.7 描述行的职责

轻量游戏里，角色状态、表情和小动作大多靠文字完成，`描述：` 非常重要。

描述行是**混编在对话中的具体动作描述**，不是分镜脚本，也不是大段旁白。它主要写角色在当前对话点里的即时反应，动作范围应控制在**原地或走动不超过两步**，用于表现角色此刻的心态、表情，并补充最基本的情节信息。

### 描述行的视角

`描述:` 行出现在游戏对话框中，呈现的是**玩家第一人称感知**——玩家所见、所听、所闻。

因此：

- **不能以玩家为主语描述玩家自身的动作**。「描述：玩家靠近，轻拍大黄侧腹」等于从第三人称叙述玩家的操作——玩家不需要被告知自己做了什么，这在对话框里毫无意义。
- 玩家的**主动意图**可以改用 `玩家："..."` 的内心独白呈现；随后的**感官反馈**再用 `描述:` 写玩家看到 / 听到的结果。

```text
不推荐：
描述：玩家靠近，轻拍大黄侧腹
描述：大黄哼了一声

推荐：
玩家："得把他叫起来问问。"
描述：轻拍大黄
描述：大黄哼了一声
```

注意第二个写法里「描述：轻拍大黄」仍是省略主语的动作提示（类似舞台指示），语义上是"玩家轻拍大黄"——但句子里没有"玩家"这个词，读起来像是画面里正在发生的事，不像旁白替玩家报告操作。**省略主语比写出「玩家」更自然。**

### 「玩家」这个词不应出现在描述行中

**「玩家」既不作主语，也不作宾语。** 描述行呈现的是玩家眼中正在发生的事，玩家本身不是被描述的对象。

```text
不推荐：
描述：蛙缓缓转头，第一次看向玩家
原因：「看向玩家」中「玩家」作宾语——描述行的视角本身就是玩家，不需要也不应该从外部指称「玩家」。

推荐：
描述：蛙缓缓转头，目光第一次落过来
```

---

### 描述行应该承担：

- 原地小动作：抬头、低头、坐下、站起、后退半步、压住某物。
- 情绪表情：停顿、尾巴弹动、视线飘开、立刻收住、沉默。
- 轻量信息提示：看向墙缝、看向屋顶、爪子压住鱼、钥匙晃了一下。
- 基本情节补充：角色刚刚被晃下树、二层窗传来一声轻响、门被从里面推开。
- 感官细节：一阵带哨音的鼾声、苹果发酵的酸甜气味、刺鼻的薄荷味。

### 描述行不应该承担：

- 大段心理分析。
- 复杂镜头设计。
- **走位与方向性动作**（转身背对、大幅走位、离场等）：轻度游戏动物 3D 动画简单，只写原地或极小范围的动作，避免穿帮。
- **场景与环境陈设**（道具摆放、背景细节、登场环境）：交给美术，不写入描述行。若需提示策划意图，改用 `〔系统注〕`。
- 需要额外关卡 / 演出资源才能实现的设计。
- 玩家尚不该知道的解释。
- **以"玩家"为主语或宾语的描述**（见上方"视角"说明）。

好的描述行要**精简但生动**：让玩家看见角色当下的反应，顺便读出一点线索，而不是让旁白讲答案。

例：

```text
推荐：
描述：黑猫停顿，视线短暂飘向红顶屋墙缝方向，很快收回

不推荐：
描述：黑猫想起老鼠偷走薄荷鱼的那天下午，怒火中烧，决定让玩家去找老鼠算账
```

## 16.8 性格表达

NPC 的性格要通过固定的说话习惯和反应方式持续体现。

写脚本前先列清楚：

- 自称 / 对玩家称呼。
- 口头禅或标志句式。
- 情绪破防点。
- 嘴硬方式。
- 对其他角色的固定偏见。
- 什么情况下会失态，失态后如何找回体面。

例：黑猫自称"本喵"，永远傲慢嘴硬；真正的软化不通过台词承认，而通过行为表现：没有离开、尾巴停住、爪子压住薄荷鱼、最终起身。

### 跨角色称谓的归属

每个绰号、外号、蔑称都有一个**发明者**，写在角色详档的"对他人称呼"里。

规则：

- **发明者**直接使用该称呼，无需标注来源。
- **其他 NPC 引用时**，必须体现"复述 / 模仿"语气，让玩家感受到"这是别人说的，这个 NPC 在借用或嘲讽"。
- 不得让不同 NPC 各自独立使用同一个称呼，仿佛每个人都各自想到了这个词——否则来源模糊，也会削弱称呼本身的喜剧感。

例：乌鸦叫大黄「地毯」（乌鸦是发明者）。黑猫引用时不直接叫"地毯"，而是冷冷说"……地毯"，再补一句"那只鸟叫的，本喵只是觉得贴切"——复述语气把来源和黑猫的性格都交代清楚了。

## 16.9 信息释放节奏

每段对话只承担一个主要推进目标。

常用节奏：

1. 玩家递事实。
2. NPC 嘴硬 / 否认 / 嘲讽。
3. 玩家补一个证据或观察。
4. NPC 露出破绽。
5. 玩家顺势追问。
6. NPC 给出有限信息或提出交换条件。

不要一次把全部答案讲完。根据 NPC 的信息策略区别对待：

- 先收钱 / 达成条件，再给情报（交易）。
- 可以给真信息，但用自己的误读包起来（误导）。
- 让玩家主动问出来，不要让 NPC 主动自爆。
- 终局信息保留到漫画收束或最终场景。

**另外**：检查相邻两段是否在传达同一核心信息——如果是，合并成一段。重复信息拖节奏，也让玩家感到被说教。

## 16.10 菜单与锁条件

菜单不是罗列所有话题，而是表现玩家当前能做出的有效行动。

菜单项出现前要问：

- 玩家是否已经获得足够证据？
- 这个选项会不会提前泄露结论？
- 这个选项是否会误导玩家以为已掌握未掌握的信息？

锁条件写在系统注里：

```text
〔菜单〕
**A** → "蛋的线索指向红顶屋里面。" → 进入案情汇报线
**B** → "谷仓角落那里……那个草窝，是你的吧。" → 进入薄荷鱼线
*〔B 选项需持【谷仓午睡点】+【烧焦稻草 / 皮毛】，否则不出现〕*
```

硬必信息可以合并成一段汇报；可缺失信息要做卡点，并给玩家明确但不剧透的返回方向。

## 16.11 支线接口写法

一个 NPC 的脚本不要展开另一个 NPC 的完整对话。

推荐写法：

```text
**★ 玩家前往老鼠处询问，返回后交还薄荷鱼**

> 〔见老鼠兄弟-对话脚本 · 薄荷鱼专线〕
> 玩家完成交易后取得关键信息 / 道具，再返回本 NPC。
```

本角色文档只写：

- 为什么玩家会去找对方。
- 从对方那里回来后，本角色如何反应。
- 本角色状态如何变化。

对方 NPC 的收费顺序、台词风格、交易细节，放在对方自己的对话脚本里。

---

## 16.12 二周目 / 回访写法

二周目对话不是继续主线，而是情绪收束。

主要功能：

- 让 NPC 对真相有新的态度。
- 回收前文口头禅或误会。
- 提供轻量彩蛋和性格余味。
- 不再承担硬必推进。

写法上可以更短、更松，但仍要保持角色嘴脸，不要突然变成温柔旁白。

**回访时机的可控性**：如果无法锁定玩家回访时机（如离场后随时可以再入，不经任何触发），该段回访只能做**等权重轮播的轻量闲聊**，不能承担新线索、新节点或推进性信息。此类回访不需要菜单，不需要锁条件，只需保持 NPC 性格即可。

---

## 16.13 脚本自检清单

每写完一个 NPC 脚本，按以下五组检查：

**台词依据**

- 玩家每一句是否都有线索、场景观察或 NPC 行为作为依据？
- 不可交互状态（跳跃 / 攀爬 / 过场）是否误加了玩家台词？
- NPC 是否只说自己此刻能知道的事？是否提前泄露了终局信息？
- 同一 NPC 在同一段是否出现自相矛盾的台词？

**描述与格式**

- 描述行是否包含场景陈设、环境描写，或方向性动作（背对、大幅走位）？
- 描述行中是否出现了「玩家」这个词（包括作宾语的情况）？
- 说话方是否使用角色本名（或正确伪装名），而非地点名、场景名？
- 描述行是否足够承担轻量演出（有没有漏写关键动作）？

**结构与节奏**

- 相邻两段是否在传达重复信息？是否可以合并？
- 菜单项是否都有明确前置条件，格式是否统一？
- 回访触发时机是否可控？不可控时是否已简化为轻量轮播？

**角色一致性**

- 角色口吻全程是否稳定？
- 所用绰号是否属于本角色认知范围？引用他人称呼时是否有复述语气？
- 玩家是否有捧哏、误会、追问，而不是只做信息复读机？
- 伏笔是否有回收？回收是否由玩家主动触发比 NPC 自爆更好？

**跨文档一致性**

- 支线是否只写接口，没有偷写别的 NPC 的完整对话？
- 交易信息是否遵守"先付出 / 达成条件，再给情报"？
- 是否同步检查角色详档、`10-NPC主时间线.md`、`13-玩家线索与交互点总表.md`？
- 树状脚本中的变量名是否与 `17-全局游戏状态变量.md` 一致？新增变量是否已先登记？
